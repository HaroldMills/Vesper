Process and thread shutdown
---------------------------

I asked Gemini 3 Pro (via GitHub Copilot in VS Code) the following question:

    I'm writing a Python application that comprises several processes,
    including a main process and some subprocesses that it creates. The
    subprocesses can themselves start additional processes, and any of
    the processes can have multiple threads. The main process is responsible
    for shutting down the application in an orderly fashion by stopping
    and joining each of its subprocesses and threads. Each of the join
    operations times out after some period. If a subprocess or thread is
    still alive after the timeout period, I would like for shutdown to
    continue, and for the program to exit even though the subprocess or
    thread did not stop. What is the best way to ensure this? Do the
    subprocesses and threads all need to be deamonic, or is there some
    better approach?

and it responded:

For your specific architecture, you cannot make the subprocesses
daemon=True.

The Python multiprocessing module enforces a strict rule: daemonic
processes are not allowed to create children. Since your subprocesses
launch their own subprocesses, they must be non-daemonic.

Here is the best approach to ensure a clean shutdown that forces an
exit if things get stuck:

1. Threads: Use daemon=True

For threads, making them daemonic is usually the right choice for this
requirement.

    * Why: You cannot easily "kill" a stuck thread in Python.
    * Behavior: If you set thread.daemon = True, the Python process
    will exit immediately as soon as the main thread (and any valid
    non-daemon threads) are done, unceremoniously killing the daemon
    threads.
    * Strategy: Try to join(timeout) them first for a clean shutdown.
    If the timeout expires, just proceed. Since they are daemonic,
    they won't block the process from exiting.

2. Processes: Use daemon=False with Explicit Termination

Since your processes are creating children, they must be daemon=False.
This means your main process will hang indefinitely at exit if you
don't ensure they are stopped.

The Strategy:

    * Signal: Tell the process to stop (e.g., set an Event or send
    SIGTERM).
    * Wait: Call process.join(timeout).
    * Force: If process.is_alive() returns True after the timeout, call
    process.terminate() (or process.kill() for a harder kill).

Example Implementation

Here is a robust pattern for your shutdown method:

    import time

    def shutdown(self):

        # 1. Signal all subprocesses to stop (implementation depends on your code)
        self._signal_stop_event.set() 
        
        # 2. Join subprocesses with timeout
        for p in self._subprocesses:
        
            p.join(timeout=2.0) # Wait up to 2 seconds
            
            if p.is_alive():
            
                # 3. Force kill if it didn't listen
                print(f"Process {p.name} did not stop gracefully. Forcing termination.")
                p.terminate() 
                
                # Optional: Allow a brief moment for cleanup after SIGTERM
                p.join(timeout=0.1) 
                
                # If still alive (e.g. stuck in system call), use SIGKILL (Python 3.7+)
                if p.is_alive():
                    p.kill()

        # 4. Threads (assuming they are daemon=True)
        # You can try to join them, but if they time out, you can just return.
        # The interpreter will kill them on exit.

A Note on "Grandchild" Processes

Since your subprocesses start their own processes, simply killing the
parent subprocess (process.terminate()) might detach the grandchildren,
leaving them running as "orphaned" processes on your OS.

To fix this, your Subprocesses need a similar cleanup routine:

They should catch SIGTERM (which process.terminate() sends).
In the signal handler or finally block, they should terminate() their
own children before exiting.


Multiprocessing and threading queue shutdown
--------------------------------------------

I asked Gemini 3 Pro (via GitHub Copilot in VS Code) the following question:

    What must one do to shut down a Python multiprocessing queue in an
    orderly fashion? How about a Python threading queue?

and it responded:

To shut down Python queues in an orderly fashion, the steps differ
significantly between multiprocessing.Queue and queue.Queue (threading)
because of how they handle data buffering.

1. multiprocessing.Queue

For a multiprocessing.Queue, "shutting down" primarily concerns the process
that put data into the queue. When you put data into a generic mp.Queue,
a background thread (the "feeder thread") starts up to buffer that data to
the underlying pipe.

To shut it down cleanly without errors (like BrokenPipeError) or hanging
processes:

    1. Stop putting data: ensure no new data is being added.

    2. Close the queue: Call q.close(). This indicates that no more data
       will be put into this queue by the current process.

    3. Join the feeder thread: Call q.join_thread(). This blocks until
       the background feeder thread has flushed all buffered data to the
       pipe.

    # In the process that was WRITING to the queue:
    q.close()
    q.join_thread()

Note: If a process is only reading from the queue, it generally does not
need to call join_thread(), but calling close() is still good practice to
release resources.

2. queue.Queue (Threading)

The standard queue.Queue runs entirely in memory within a single process
and does not have a background feeder thread or a close() method.
"Shutting down" usually means signaling consumer threads to stop.

The standard pattern is:

    1. Send Sentinels: Put a special object (often None) into the queue
       to signal consumers to exit. You typically send one sentinel per
       consumer thread.

    2. Wait for completion (Optional): If you are using the
       q.join() / q.task_done() pattern to track work, call q.join() to
       wait until all items (excluding sentinels if handled correctly)
       are processed.

    # Signal shutdown
    for _ in range(num_worker_threads):
        q.put(None) # Sentinel

    # Wait for workers to finish processing existing items
    q.join() 

Summary for your RecorderProcess

I noticed in your recorder_process.py that you are already correctly
handling the logging queue in _stop_logging:

    queue = self._logging_queue
    queue.close()
    queue.join_thread()

This is correct because RecorderProcess is a writer to the logging queue.

For self.command_queue, assuming RecorderProcess only reads from it:

    * You don't strictly need to call join_thread().

    * You can just let it be garbage collected, or call close() if you
      want to be explicit.

    * The Main Process (which writes commands to this queue) is
      responsible for calling join_thread() on it if it wants to ensure
      commands are flushed before exiting.


Logging Shutdown
----------------

In an earlier version of the Vesper Recorder, I had some logging problems
at shutdown. Some messages were sometimes not logged, and I also often saw
an exception message like the following at shutdown:

    Exception in thread Thread-6 (_monitor):
    Traceback (most recent call last):
        File "/Users/harold/miniconda3/envs/vesper-dev/lib/python3.11/threading.py", line 1045, in _bootstrap_inner. 

One of my main goals in the newer version of the recorder is to handle
recorder shutdown better, for example by stopping and joining all
subprocesses and threads. With this in mind, I asked ChatGPT 5 (via GitHub
Copilot in VS Code) about the above error message, and it responded (in part):

    It’s the logging QueueListener’s background thread dying during
    interpreter shutdown. You start a QueueListener but never stop it.
    Its internal _monitor thread is still reading from the
    multiprocessing.Queue when handlers/objects are being torn down,
    so you see a traceback like “Exception in thread Thread-… (_monitor)”.

I then asked the followup question:
    
    I'm writing a Python application whose main process will create
    multiple subprocesses. Each of the subprocesses will use the logging
    module of the Python Standard Library to perform logging, sending log
    messages to a multiprocessing logging queue managed by the main process.
    Do the subprocesses need to call the logging.shutdown function when they
    complete, say by registering an appropriate function via the atexit
    module, or does only the main process need to call logging.shutdown?

and got this response:

    Short answer: only the main process should call logging.shutdown.
    Subprocesses should not.

    Guidance:

    * Main process

      - Owns the QueueListener and any real handlers (file/stream).

      - Stop the listener and then call logging.shutdown after all
        children exit.
        
    * Subprocesses

      - Use only a QueueHandler pointing at the shared multiprocessing.Queue.

      - Do not stop the listener and do not call logging.shutdown (it’s
        unnecessary and can be misleading).

      - Clean up per-process resources: remove/close the QueueHandler and
        close/join the process’s end of the queue so its feeder thread exits.

Example cleanup:


    # child process cleanup
    import logging

    def cleanup_child_logging(logging_queue, queue_handler):
        root = logging.getLogger()
        try:
            root.removeHandler(queue_handler)
            queue_handler.close()
        finally:
            # close this process's handle to the queue so the feeder thread exits
            logging_queue.close()
            logging_queue.join_thread()


    # main process cleanup
    import logging
    from logging.handlers import QueueListener

    def cleanup_main(listener: QueueListener):
        try:
            listener.stop()   # drains queue and stops background thread
        finally:
            logging.shutdown()  # flush/close file/stream handlers


    Key points:

    * Don’t call listener.stop() from children.

    * Closing/joining the child’s queue handle prevents shutdown tracebacks
      from feeder/monitor threads.

I later asked ChatGPT 5 the following question:

    Could you please explain how the Python Standard Library classes
    logging.handlers.QueueListener and logging.handlers.QueueHandler
    work together to safely log messages from multiple processes? I'm
    particularly interested in any extra threads that are involved,
    ensuring that they terminate gracefully when a program shuts down.

Its response:

    Here’s how QueueHandler and QueueListener work together for safe,
    multi-process logging, and what threads are involved.

    Concept:

    * Child/subprocess side (producer):

      - Attach a logging.handlers.QueueHandler to the process’s root logger.

      - QueueHandler.enqueue() puts LogRecord objects onto a
        multiprocessing.Queue.

      - QueueHandler.prepare() makes the record picklable (formats args,
        converts exc_info, etc.), so it can cross process boundaries.

    * Main process side (consumer):

      - Create the multiprocessing.Queue once in the main process.

      - Start a logging.handlers.QueueListener with that queue and the real
        handlers (FileHandler, StreamHandler, etc.).

      - QueueListener runs a dedicated background thread that pulls records
        from the queue and emits them via the configured handlers.

    Threads involved:

    * In each producer process:

      - multiprocessing.Queue uses a feeder thread to serialize and write
        objects to its pipe. That feeder thread must be allowed to exit
        cleanly.

    * In the main/consumer process:

      - QueueListener starts one background thread (its “monitor” thread)
        that loops on queue.get(), dispatching records to handlers until
        it sees a sentinel and exits.

    Graceful shutdown:

    * Subprocesses (producers):

      - Do not call logging.shutdown().

      - Remove and close the QueueHandler from the process’s root logger.

      - Close and join the process’s end of the multiprocessing.Queue to
        let its feeder thread exit:

            queue.close()
            queue.join_thread()

    * Main process (consumer):

      - Stop the QueueListener to terminate its monitor thread and drain
        remaining records:

            listener.stop()     # posts a sentinel and joins the thread

      - Then flush/close real handlers:

            logging.shutdown()  # optional to call explicitly; it’s also registered via atexit, but explicit is safer in controlled shutdowns)

    Recommended pattern (minimal example):

        import logging
        from logging import Formatter, FileHandler, StreamHandler
        from logging.handlers import QueueHandler, QueueListener
        import multiprocessing as mp
        import atexit

        def setup_main_logging():
            q = mp.Queue()

            console = StreamHandler()
            console.setFormatter(Formatter('%(asctime)s %(processName)s %(levelname)s %(message)s'))

            file = FileHandler('app.log', encoding='utf-8')
            file.setFormatter(Formatter('%(asctime)s %(processName)s %(levelname)s %(message)s'))

            listener = QueueListener(q, console, file)
            listener.start()

            # Ensure listener stops before logging.shutdown runs.
            atexit.register(listener.stop)
            return q, listener

        def setup_child_logging(q, level=logging.INFO):
            root = logging.getLogger()
            root.setLevel(level)
            qh = QueueHandler(q)
            root.addHandler(qh)
            return qh

        def cleanup_child_logging(q, qh):
            root = logging.getLogger()
            try:
                root.removeHandler(qh)
                qh.close()
            finally:
                # Let the feeder thread exit cleanly.
                q.close()
                q.join_thread()

        def worker(q):
            qh = setup_child_logging(q)
            try:
                logging.info('Hello from child')
            finally:
                cleanup_child_logging(q, qh)

        if __name__ == '__main__':
            q, listener = setup_main_logging()

            procs = [mp.Process(target=worker, args=(q,)) for _ in range(2)]
            for p in procs: p.start()
            for p in procs: p.join()

            # Controlled shutdown (optional; atexit will also run):
            listener.stop()
            logging.shutdown()

    Key takeaways:

    * QueueHandler only enqueues records; real formatting/output happens
      in the main process.

    * QueueListener runs one background thread to consume and dispatch
      records.

    * Clean shutdown requires:

      - Children: remove/close QueueHandler, then close/join their queue
        handle (stops feeder threads).

      - Main: listener.stop() (stops monitor thread), then logging.shutdown()
        to flush/close handlers.

    * Relying solely on atexit for logging.shutdown is usually fine,
      but you still must stop the QueueListener explicitly to avoid
      shutdown tracebacks and lost records.

Further questions clarified that if the main process wants to log messages,
it should also use a QueueHandler to send them to the logging queue, and
perform the same cleanup steps for its QueueHandler as child processes.


Here's the result I got from googling "python QueueFeederThread":

The QueueFeederThread is an internal, non-daemonic thread used by the
Python multiprocessing.Queue to transfer data from a buffer in the
current process to the underlying inter-process pipe. It is not a
class you explicitly create, but rather a background component managed
automatically by the multiprocessing module.

Purpose

The main purpose of the QueueFeederThread is to prevent put()
operations from blocking indefinitely if the underlying pipe's buffer
is full. It runs in the background, continuously moving items from the
process's internal buffer to the shared pipe, allowing the main process
to continue adding items to the buffer without waiting for the consumer
processes to retrieve them immediately. 

Common Issues and Solutions

The QueueFeederThread is a common source of deadlocks or hanging
processes, especially during program shutdown, because the parent
process waits for it to finish (join).

* Hanging Processes at Shutdown: If the consumer process(es) do not
empty the queue before the producer process tries to exit, the
QueueFeederThread will wait indefinitely for the pipe to drain,
causing the process to hang.

* Exceptions during Interpreter Shutdown: Sometimes, a TypeError
exception ('NoneType' object is not callable') related to this thread
may appear during interpreter shutdown, often a symptom of underlying
race conditions in older Python versions. 

Best Practices for Graceful Shutdown

To ensure a graceful shutdown and prevent hanging, the following
methods are recommended:

* Empty the Queue: Ensure all items are removed from the queue by
consumer processes before the program terminates.

* Use Sentinel Values: A common pattern is to put a special "sentinel"
value (like None) into the queue when production is complete. Consumer
processes, upon receiving this sentinel, know to finish their tasks and
exit.

* Call cancel_join_thread(): A process can call the cancel_join_thread()
method on the queue object to avoid waiting for the feeder thread to join.
Note, however, that any data still in the internal buffer might be lost.

* Consider multiprocessing.SimpleQueue: If a background feeder thread is
not desired (e.g., for write-only scenarios), the
multiprocessing.SimpleQueue can be used, as it does not have a feeder
thread and put operations block if the buffer is full.

* Use a Manager-backed Queue: Using multiprocessing.Manager().Queue()
runs the queue in a separate server process, which avoids some of the
feeder thread issues within child processes, though it adds a slight
overhead.
